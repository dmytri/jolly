## Governing principle: live-by-design, never mock/fake

The suite runs against **real services** in a production-shaped test env (the `JOLLY_*` `.env` credentials), every tier including `@logic`. Never mock/fake (no fake CLIs, dummy creds, `.invalid` endpoints). Creating real resources is expected; safety is **harmless-by-design** — namespace every created resource + idempotent teardown + never touch what the run did not create (AGENTS.md). Produce failures from real bad input. The only doubles are inline `@exceptional-double`s for conditions the real env can't produce on demand (current set: `ENVIRONMENT_LIMIT_REACHED`, the unverifiable-endpoint "stored, not verified" path, and the device-grant human-approve via the local fake auth host). Enforced by 026's `@property` "no forbidden double". A persistently-skipping `@sandbox` scenario is un-verified, not done ([[skip-mask-sandbox-unverified]]).

## Change-impact selection + durable green ledger (dk, 2026-07-09) -- BIG, upstream Shipshape

dk's principle: a scenario that was EVER green NEVER needs to run again, EXCEPT (1) it is in the blast radius of a change made since it went green, or (2) a full harbour/boundary inspection -- whose greens then PERSIST as noted. This is test-impact analysis + a durable result cache. It is the execution model a verification-driven system should have; re-running a known-green outside a change's blast radius is provably wasted work. Shipshape already leans this way ("focused checks over full tier runs; full runs are boundary checks not the inner loop"; wake carries yesterday's weather). This pushes it to its sound conclusion: derive the focused set from blast radius, and CACHE greens instead of re-running.

**Soundness condition (the whole game):** "green at C => green now" holds iff each scenario's input closure is soundly computed. `closure = src call/import closure executed + step defs + shared support + config/resolved deps (cucumber.js, tsconfig, lockfile) + referenced assets`. Cache key = hash of that closure; change-set intersect closure => re-run, else inherit green. Machinery already present: per-scenario coverage (the `coverage` command: which src lines each scenario hits) gives the mechanical reverse `line -> scenarios` map; `@planks` is a complementary behavioural index. Over-approximate (err toward re-running) and it stays sound.

**The one place the rule breaks -- and it is exactly this project (live-by-design):** sound for `@logic` (local/deterministic -- the big safe win, ADOPT). UNSOUND for `@sandbox`/`@eval`, because their closure includes a mutable, unhashable input: the real external service (Saleor Cloud / Vercel / Paper), which can change with zero change on our side -- and catching that upstream drift is the entire reason those tiers exist (worked case in these notes: "Paper main build break -- re-verify once Paper fixes main"; nothing in Jolly changed, scenario flips red; local blast-radius would skip it forever). So live tiers need a THIRD exception: freshness / upstream-pin. A live green is cached with a timestamp + upstream version/API pins; blast radius OR staleness re-runs it; a boundary/harbour run refreshes it.

**"Persist as noted" = a per-scenario green ledger** in the wake: last-green commit, closure hash, and (live tiers) timestamp + version pins. Coherent with Shipshape's wake -- a richer hash-keyed weather record used for SELECTION, not just concurrency priors. Note: today's full all-tier run IS a boundary/harbour inspection -- precisely the event that seeds this ledger, not a per-iteration cost.

**Caveats (honest):** (a) flakiness is the enemy of caching -- a flaky green (PTY `@logic` flakiness; live `@eval`) is not a stable green; caching it hides an intermittent bug; the model rewards the determinism Shipshape already demands. (b) Tensions Article 6 ("green means the real thing happened THIS run"): reconciled because a cached green is exactly as strong as its closure is sound -- airtight for `@logic`, and the freshness exception is mandatory (not optional) for live tiers.

### Note for upstream Shipshape (third lesson, the biggest)

> Shipshape says the inner loop is focused and full runs are boundaries, but leaves selection manual (watchbill) and does not define a durable green cache. Propose a change-impact selection model: (1) compute each scenario's input closure from per-scenario coverage + trace annotations + config/deps; (2) maintain a durable per-scenario green ledger in the wake keyed on the closure hash; (3) the inner loop runs only scenarios whose closure intersects the change set since their last recorded green; (4) boundary/harbour runs refresh the ledger and their greens persist. CRUCIAL tier split: for deterministic/local tiers the cache is sound and "ever green -> never re-run unless in blast radius" holds; for LIVE-service tiers the external service is a mutable unhashable input, so add a freshness/upstream-pin exception -- a live green is valid only within a freshness window or until pinned upstream versions move, because those tiers exist to catch drift the project did not cause. This makes verification latency (which in a spec-driven system IS iteration latency) scale with change size, not suite size.

### Note for upstream Shipshape (second lesson)

Complements the "scantling vs. scenario-outline" lesson above. For a SLOW `@sandbox`/real-service scenario, the first question is NOT "can a scantling replace it" but "does it need real execution":

> A slow real-service scenario is not automatically a scantling candidate. First ask whether the behaviour needs real execution (offline install actually lands files; a headless installer does not silently no-op -> yes, real). If yes, the lever is the Verification agreement's minimal-sufficient-fixture / reuse clauses, NOT a scantling: exercise the behaviour through the SMALLEST real resource that proves it. "Install the full production set" to assert a per-installer property (non-interactivity, on-disk verification) is over-provisioning -> scope to a representative resource. A scenario that provisions a full set under a condition where most of it cannot succeed (offline install of remote refs) is spending wall-clock on doomed work it does not assert -> scope to what it asserts. A scantling may still ADD a cheap structural guard for the packaging/shipment facet, but it does not replace the runtime proof. Jolly worked example: `007` two `@sandbox` installs at ~150s each -> right-size to 1-2 representative skills + scope the no-network case to the bundled skill; keep a real full-set install only at harbour cadence as the upstream-ref integration check.
